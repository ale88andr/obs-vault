Ключевым компонентом модели распараллеливания в Go служит **горутина**, которая   является минимальной исполняемой сущностью в этом языке. В Go вообще все выполняется в виде горутин либо явно, либо по задумке. Каждая выполняемая программа Go имеет по крайней мере одну горутину, которая используется для запуска функции `main()` пакета `main`. Каждая горутина выполняется в единственном потоке ОС в соответствии с инструкциями планировщика Go, который и отвечает за выполнение горутин. Планировщик ОС не определяет, сколько потоков нужно создавать среде выполнения Go. Среда создаст достаточное количество потоков с целью гарантировать, что для запуска   Go-кода доступно `GOMAXPROCS` потоков.

Однако горутины не могут напрямую взаимодействовать друг с другом. Обмен данными в Go осуществляется с использованием либо **каналов**, либо **общей памяти**. Каналы действуют как клей, соединяющий несколько горутин. Помните: горутины могут обрабатывать данные и выполнять команды, но не могут напрямую взаимодействовать друг с другом. Тем не менее им доступны другие способы, включая каналы, локальные сокеты и общую память. В то же время каналы не могут обрабатывать данные или выполнять код, но могут отправлять данные в горутины, получать данные или иметь какое-то иное специальное на значение.

Объединяя несколько каналов и горутин, вы получаете возможность создавать  потоки данных, которые в терминологии Go также называются конвейерами. 

### Процессы, потоки и горутины

#### Процессы

- **Программа** — это просто файл на диске (например, `.exe` или `.elf` в Linux), в котором записаны инструкции для компьютера. Пока она лежит на диске, она «мертвая» — это набор команд.
- **Процесс** — это та же программа, но уже запущенная. Операционная система берёт файл, загружает его в память и начинает выполнять инструкции.

Когда программа превращается в процесс, ОС добавляет к ней всё необходимое для работы:
- выделяет память;
- создаёт таблицу открытых файлов;
- хранит данные пользователя;
- управляет другими ресурсами (например, сетевыми соединениями).

Запомнить просто:

- **Программа** — это рецепт на бумаге.
- **Процесс** — это блюдо, которое начали готовить по этому рецепту: есть продукты (память, файлы), есть сам процесс приготовления (выполнение инструкций).

Особенности:

- Каждый процесс имеет собственное **адресное пространство** — память, недоступную другим процессам напрямую.
- Межпроцессное взаимодействие (IPC) требует специальных механизмов: сокеты, каналы, очереди сообщений, разделяемая память.
- Создание нового процесса (например, через `fork` в Unix) — операция тяжёлая: копируется контекст, таблицы файловых дескрипторов и т. д.

#### Потоки

- **Процесс** — это запущенная программа.
- Внутри процесса может быть **один или несколько потоков**.
- **Поток** — это отдельная последовательность выполнения команд внутри процесса.

Каждый поток:

- выполняет свои инструкции независимо от других потоков;
- имеет собственный **стек** (рабочую область памяти для временных данных и вызовов функций);
- делит общую память и ресурсы процесса (файлы, данные, переменные).

Главное назначение потока: Позволить программе выполнять несколько задач одновременно. Например:

- один поток обрабатывает ввод пользователя,
- другой загружает данные из сети,
- третий выполняет вычисления.

Таким образом:

- **Процесс** — это сама программа с ресурсами.
- **Поток** — это «исполнитель команд» внутри процесса, который работает параллельно с другими потоками, но использует общие ресурсы.

Особенности:

- Потоки внутри одного процесса **делят общее адресное пространство**, что упрощает обмен данными, но повышает риск ошибок (например, гонки данных).
- У каждого потока есть собственный **стек** и **счётчик команд**, но общие ресурсы процесса (файлы, память) доступны всем.
- Создание потока легче, чем процесса, но всё же требует взаимодействия с ОС.

#### Горутины

- В отличие от потоков ОС, горутины управляются **планировщиком Go** (runtime scheduler), который распределяет их по доступным потокам ОС.
- Начальный стек горутины очень мал (несколько килобайт), и он динамически растёт при необходимости.
- Переключение между горутинами происходит быстрее, чем между потоками, так как это управляется самим рантаймом Go, а не ядром ОС.
- Для синхронизации и обмена данными используются **каналы** — встроенный механизм Go, обеспечивающий безопасное взаимодействие без явных блокировок.

### Горутины

Вы можете определить, создать и выполнить новую горутину, используя ключевое слово `go`, за которым следует имя функции или анонимная функция. Ключевое слово `go` немедленно возвращает выполнение из вызова функции, в то время как функция начинает работать в фоновом режиме в виде горутины, тогда как остальная часть программы тоже продолжает свое выполнение.

> Вы не можете контролировать или делать какие-либо предположения о порядке, в котором будут выполняться ваши программы, поскольку это зависит от планировщика ОС, планировщика Go и загрузки ОС.

##### Создание горутины

```go
func main() {
	// Так запускается анонимная функция в качестве горутины
	// (10) в конце — это способ, с помощью которого передается 
	// параметр анонимной функции. Анонимная функция выше просто 
	// выводит значение на экран.
	go func(x int) {
		fmt.Printf("%d ", x)
	}(10)
	
	go printme(15)
	
	time.Sleep(time.Second)
	fmt.Println("Exiting...")
}
```

Как правило, функции, которые выполняются как горутины, не возвращают никаких значений напрямую. Обмен данными с горутинами происходит с помощью общей памяти, каналов или какого-либо другого механизма.

##### Ожидание завершения горутин

Недостаточно создать несколько горутин‚ нужно еще дождаться их завершения до завершения функции `main()`.

Процесс синхронизации начинается с определения переменной `sync.WaitGroup` и использования методов `Add()`, `Done()` и `Wait()`.

Если вы посмотрите на исходный код Go-пакета `sync`, а точнее на файл `waitgroup.go`, то увидите, что тип `sync.WaitGroup` — это не что иное, как структура с двумя полями:

```go
type WaitGroup struct {
	noCopy noCopy
	state1 [3]uint32
}
```
