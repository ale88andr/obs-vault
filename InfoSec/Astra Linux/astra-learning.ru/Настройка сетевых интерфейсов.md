## Сети на основе стека TCP/IP

> Стек протоколов TCP/IP — это иерархически организованный набор сетевых протоколов, достаточный для организации взаимодействия узлов в сети.

Для описания технических спецификаций сетевых протоколов и сервисов используются стандарты **Request for Comments (RFC)**. Согласно **rfc1122**, стек интернет-протоколов состоит из четырёх уровней
- прикладной уровень (Application Level);
- транспортный уровень (Transport Level);
- межсетевой уровень (Internet Level);
- канальный уровень (Link Layer).

**Прикладной уровень** протоколов интернета — это уровень, через который пользовательские программы взаимодействуют с сетью. Протоколы прикладного уровня разделяются на две категории:
- пользовательские протоколы (`SSH`, `Telnet`, `FTP`, `SMTP`), которые предоставляют услуги непосредственно пользователям;
- протоколы поддержки (`DNS`, `BOOTP`, `SNMP`), которые обеспечивают общие системные функции.

Протоколы **транспортного уровня** используются для обеспечения взаимодействия сетевых приложений. На транспортном уровне определяются адреса сетевых приложений в рамках хоста — порты. В настоящее время существует два основных протокола транспортного уровня:
- **TCP** (Transmission Control Protocol) — предназначен для надёжной и гарантированной передачи (доставки) данных между хостами в компьютерных сетях с коммутацией пакетов и между сетями через промежуточные системы;
- **UDP** (User Datagram Protocol) — используется для передачи данных в режиме реального времени, таких как потоковое видео и аудио. Протокол не гарантирует надёжную доставку данных и порядок их поступления.

**Межсетевой уровень** отвечает за маршрутизацию пакетов между сетями, т. е. за адресацию `IP`, с помощью которой возможно идентифицировать и локализовать устройства в сети Интернет. Также он определяет, какой путь должны проходить пакеты от отправителя к получателю. Основной протокол межсетевого уровня — это Internet Protocol (`IP`).

**Канальный уровень** предназначен для обмена данными между соседними устройствами внутри одного сетевого сегмента (локальной сети). Для связи с непосредственно подключёнными к нему сетями хост должен поддерживать коммуникационный протокол, используемый для обмена данными с сетью.

## IP-адреса

Каждому сетевому интерфейсу должен быть присвоен **IP-адрес**. Его можно назначить статически, динамически, а также с помощью службы **DHCP**.

Существует две версии IP-протокола, которым соответствуют разные форматы адресов:
- `IPv4` (Internet Protocol version 4) — четыре байта. Записывается в виде четырёх десятичных чисел, разделённых точками (например, `192.168.1.1`), где каждое число может варьироваться от `0` до `255`.
- `IPv6` (Internet Protocol version 6) — шестнадцать байтов (`128` бит). Записывается в виде восьми чисел в шестнадцатеричном формате, разделяемых двоеточиями (например, `2001:0db8:85a3:0000:0000:8a2e:0370:7334`).

Принцип работы `DHCP`. Этот процесс называют **DORA**:
- `Discovery` (поиск) — клиент отправляет широковещательный запрос на поиск DHCP-сервера;
- `Offer` (предложение) — сервер предлагает клиенту IP-адрес и другие параметры;
- `Request` (запрос) — клиент подтверждает выбор предложенного адреса;
- `Acknowledgement` (подтверждение) — сервер подтверждает выделение адреса.

**Первый** адрес в сети — адрес самой сети. **Последний** адрес — адрес всех хостов в данной сети (широковещательный адрес).

Какой раздел IP-адреса относится к идентификатору сети, а какой — к идентификатору хоста, зависит от класса адреса или маски подсети, если используется подсетевое разделение.

**Маска сети** — это числовая комбинация, используемая для разделения IP-адреса на две части: **идентификатор сети** (`Network ID`) и **идентификатор хоста** (`Host ID`). Например, в домашних или малых офисных сетях обычно применяется маска `255.255.255.0`. Это означает, что первые три числа в IP-адресе определяют сеть, а последнее число — конкретный хост внутри этой сети.

Маска сети используется для подсетевого разделения, что позволяет разбивать крупные сети на более мелкие подсети. Это обеспечивает гибкое управление IP-адресами, повышает безопасность и облегчает маршрутизацию в сложных сетевых структурах.

## Протокол IPv4

`IPv4` — это четвёртая версия интернет-протокола, которая используется для передачи данных в сети Интернет. Он присваивает каждому устройству уникальный 32-битный IP-адрес, состоящий из четырёх групп чисел от `0` до `255`, разделённых точками (например, `192.168.1.1`). Хотя этот протокол всё ещё широко используется, его адресное пространство почти исчерпано.

В сетях `IPv4` поддерживаются три типа адресов:
- `Unicast` — единичный адрес, который соответствует единственному устройству в сети. Unicast-сообщения будут доставлены только этому устройству;
- `Multicast` — адреса для группы устройств. Multicast-сообщения будут доставлены всем устройствам в группе. Обычно используется для потоковых медиа и групповой связи;
- `Broadcast` — адрес указывает на все устройства в текущей IPv4-сети. Broadcast-сообщения будут доставлены всем устройствам, входящим в эту сеть. Адрес `255.255.255.255` в `IPv4` используется для ограниченного широковещания (limited broadcast), но он не предназначен для рассылки на устройства в разных IPv4-сетях, а только для всех узлов в пределах одной и той же подсети, в которой находится источник пакета.

Адрес IPv4 состоит из двух основных частей:
- **идентификатор сети** (Network ID) — часть адреса, указывающая на конкретную сеть в интернете. Все хосты в одной и той же сети имеют одинаковый идентификатор сети;
- **идентификатор хоста** (Host ID) — часть адреса, которая указывает на конкретный компьютер (или иную машину) в сети. Идентификатор хоста уникален в пределах своей сети.

## Протокол IPv6

`IPv6` — это новая версия интернет-протокола, призванная решить проблему исчерпания адресов `IPv4`. Его спецификация описана в `RFC 8200`.

Ключевые особенности и отличия от `IPv4`:
- Гораздо большее адресное пространство: длина адреса составляет `128` бит (по сравнению с `32` битами в IPv4). Это позволяет иметь `2^128` (около `3,4×10^38`) уникальных адресов.
	>Пример записи: `2001:0db8:85a3:0000:0000:8a2e:0370:7334`. Для сокращения допускается удаление ведущих нулей в каждом хекстете и замена одной последовательности нулевых хекстетов на двойное двоеточие (::): `2001:db8:85a3::8a2e:370:7334`.
- Упрощённый заголовок — заголовок `IPv6` фиксированного размера (`40` байт), что упрощает и ускоряет его обработку маршрутизаторами. Отсутствуют поля для фрагментации (она реализована иначе), контрольная сумма заголовка (расчёт возложен на вышележащие протоколы).
- Встроенная безопасность (`IPsec`) — поддержка `IPsec` (шифрование и аутентификация на сетевом уровне) является обязательной для реализации `IPv6`, хотя её использование остаётся опциональным.
- Эффективная маршрутизация — иерархическая структура адресации и агрегация маршрутов, заложенные в IPv6, позволяют сохранять таблицы маршрутизации компактными.    
- Автоконфигурация:
	- Stateless Address Autoconfiguration (`SLAAC`) — позволяет хосту автоматически сконфигурировать свой адрес на основе префикса, рассылаемого маршрутизатором, и своего MAC-адреса. Адрес маршрута по умолчанию выдаётся маршрутизатором;
	- Stateful `DHCPv6` — аналог `DHCP` для `IPv6`, когда адреса назначаются централизованно сервером.

Типы адресов IPv6:
- **Unicast** — адрес, идентифицирующий один интерфейс. Пакет доставляется только этому интерфейсу:
- **Global Unicast Address** (`GUA`) — публичные адреса, маршрутизируемые в интернете (аналогичны публичным IPv4). Диапазон — `2000::/3`.
- **Link-Local Address** (`LLA`) — адреса, используемые для связи только в пределах одного канального сегмента сети (аналогичны `169.254.0.0/16` в `IPv4`). Все интерфейсы имеют `LLA`, сгенерированный на основе `FE80::/10`.
- **Multicast** — адрес, идентифицирующий группу интерфейсов. Пакет доставляется всем интерфейсам группы. Широковещательных (Broadcast) адресов в IPv6 нет, их функцию полностью выполняет multicast.
- **Anycast** — адрес, назначенный нескольким интерфейсам (обычно на разных устройствах). Пакет доставляется ближайшему (с точки зрения метрики маршрутизации) интерфейсу с этим адресом.

## Классы IP-адресов

Для упрощения обозначения различных размеров сетей IP-адреса исторически были разделены на пять классов:
- **класс A** — выделяется очень большим сетям (адреса в диапазоне от 1.0.0.0 до 126.255.255.255);
- **класс B** — выделяется средним и большим сетям (адреса в диапазоне от 128.0.0.0 до 191.255.255.255);
- **класс C** — выделяется небольшим сетям (адреса в диапазоне от 192.0.0.0 до 223.255.255.255);
- **класс D** — предназначен для поддержки групповых рассылок данных (адреса в диапазоне от 224.0.0.0 до 239.255.255.255);
- **класс E** — зарезервирован для будущего использования или для исследований (адреса в диапазоне от 240.0.0.0 до 255.255.255.254).

Классовая адресация является устаревшей и в современном интернете не используется. Её заменила бесклассовая адресация (CIDR).

Согласно `RFC 1700`, выделены специальные IP-адреса:
- `{<номер сети>, 0}` — обозначает всю сеть, 192.168.1.0 (для сети 192.168.1.0/24);
- `{<номер сети>, все 1}` — широковещательный адрес сети (используется только как адрес назначения), 192.168.1.255 (для сети 192.168.1.0/24);
- `{все 1, все 1}` — локальная рассылка (рассылка внутри своей сети, используется только как адрес назначения), 255.255.255.255;
- `{все 0, все 0}` — текущий хост в текущей сети (используется только как адрес источника), 0.0.0.0;
- `{127, <произвольно>}` — loopback (любой пакет, направленный на этот адрес, не достигающий передающей среды, передаётся локальному сетевому приложению), 127.0.0.1.

## IP-адреса для частных сетей

Приватные сети предназначены для хостов:
- которым не нужен доступ к публичной сети Интернет;
- которые изолированы от внешней сети шлюзом.

Понятие частных (private) сетей определяют RFC `1597` и `1918`.

Управление по присвоению номеров в интернете (IANA) зарезервировало следующие три блока IP-адресов для частных сетей
- `10.0.0.0−10.255.255.255` (одна сеть класса A);
- `172.16.0.0−172.31.255.255` (16 сетей класса B);
- `192.168.0.0−192.168.255.255` (256 сетей класса C).

## Подсети

Концепция подсети (subnet), которая позволяет разбивать сети класса `A`, `B` и `C` на более мелкие, была введена RFC `917` и `950`.

В рамках концепции подсети адрес из любой сети класса `A`, `B` или `C` состоит из трёх частей:
- номер сети;
- номер подсети;
- номер хоста.

Номер подсети и номер хоста содержатся в номере хоста исходной сети. Номер сети и номер подсети образуют вместе расширенный сетевой префикс. Для выделения сетевого префикса и номера хоста используется маска подсети.

**Маска подсети** — это 32-битовое число. Биты маски, равные `1`, выделяют биты в адресе, используемые в качестве сетевого префикса. Биты маски, равные `0`, используются для нумерации хостов.

В RFC `1009` определён механизм поддержки нескольких масок в одной сети. Механизм получил название масок подсетей переменной длины (VLSM) и позволяет разбивать сеть на подсети разных размеров.

Однако классовый подход к адресации оказался негибким и приводил к расточительному использованию IP-адресов.

Для решения этих проблем в RFC `1517`, `1518`, `1519`, `1520` был описан механизм бесклассовой междоменной маршрутизации (`CIDR` — Classless Inter-Domain Routing).

## CIDR — бесклассовая междоменная адресация

В `CIDR` нет жёсткой привязки к классам `A`, `B` или `C`. Сеть определяется исключительно комбинацией IP-адреса и маски переменной длины.

CIDR полностью использует механизм масок подсетей переменной длины (VLSM), позволяя разбивать сеть на подсети разных размеров в зависимости от потребностей.

Также CIDR позволяет объединять несколько сетей класса C в одну большую суперсеть. Это было первым шагом в борьбе с истощением адресов IPv4.

Агрегирование маршрутов — это ключевое преимущество CIDR. Несколько смежных сетей можно объединить в одну запись в таблице маршрутизации, используя общий префикс.

Например, сети `192.168.0.0/24`, `192.168.1.0/24`, `192.168.2.0/24` и `192.168.3.0/24` можно агрегировать в одну запись — `192.168.0.0/22`. Это значительно уменьшает размер таблиц маршрутизации в магистральных сетях Интернета.

Адреса в бесклассовом виде записываются как IP-адрес/длина_префикса, где длина_префикса — это количество бит в маске подсети, отведённых под сетевую часть.

> Пример. `192.168.1.0/26`. Здесь `192.168.1.0` — адрес сети, а `/26` означает, что первые `26` бит из `32` относятся к сетевому префиксу. Маска подсети в десятичном формате будет `255.255.255.192`.

Современный интернет работает исключительно на основе `CIDR`, а классовая адресация является устаревшей исторической концепцией.

## Основные протоколы стека TCP/IP

Семейство протоколов `TCP/IP` состоит из следующих основных протоколов:
- протокол определения аппаратных адресов (`ARP`, Address Resolution Protocol), выполняющий определение аппаратных (Ethernet) адресов по известным логическим (`IP`) сетевым адресам;
- межсетевой протокол (`IP`, Internet Protocol) обеспечивает доставку с одной машины на другую без дополнительной обработки данных;
- протокол управления передачей данных (`TCP`, Transmission Control Protocol), который отвечает за гарантированную доставку данных между сетевыми процессами;
- протокол пользовательских датаграмм (`UDP`, User Datagram Protocol), который обеспечивает негарантированную пересылку данных между сетевыми процессами;
- межсетевой протокол управления сообщениями `ICMP` (Internet Control Message Protocol) отвечает за низкоуровневую поддержку работы протокола IP: сообщения об ошибках, содействие в маршрутизации и т. п.

> **Датаграмма** — блок информации, передаваемый от источника к получателю (RFC `760`).

## ARP (Address Resolution Protocol)

`ARP` (Address Resolution Protocol) — это сетевой протокол, который сопоставляет IP-адрес устройства с его физическим MAC-адресом в локальной сети. Когда компьютеру или устройству в сети требуется отправить пакет на другое устройство, ему изначально известен только IP-адрес. Однако для отправки пакета на физическом уровне требуется знать MAC-адрес.

ARP предоставляет этот механизм, позволяя устройству выполнить запрос ARP, чтобы узнать MAC-адрес, связанный с IP-адресом. Это делается путём широковещательной передачи запроса ARP в сеть, и устройство с соответствующим IP-адресом отвечает своим MAC-адресом. Данная информация затем сохраняется в таблице `ARP` (кеше) для дальнейшего использования для того, чтобы избежать необходимости выполнять этот процесс каждый раз, когда требуется совершить одну и ту же трансляцию. 

Параметры для настройки сетевого модуля, реализующего протокол ARP, находятся в `/proc/sys/net/ipv4/neigh/default`. ARP-протокол **не является маршрутизируемым**, поэтому ARP-пакеты не выходят за пределы локального сетевого сегмента.

Структура ARP-пакета имеет следующий вид:
- тип оборудования (Hardware Type) — определяет тип сети, в которой используется ARP (для Ethernet оно будет иметь значение 1, для IEEE 802.x — 6);
- тип протокола (Protocol Type) — определяет протокол, используемый на уровне сети (IP — 2048, ARP — 2054);
- длина аппаратного адреса (Hardware Address Length) — длина (в байтах) физического (MAC-) адреса (для Ethernet — 6 байтов);
- длина программного адреса (Protocol Address Length) — длина (в байтах) адреса сетевого уровня (для IP — 4 байта);
- код операции (Operation Code) — значение 1 указывает на запрос ARP (ARP Request), значение 2 — на ответ ARP (ARP Reply);
- аппаратный адрес отправителя (Sender MAC Address) — MAC-адрес устройства, отправившего пакет ARP;
- IP-адрес отправителя (Sender IP Address) — IP-адрес устройства, отправившего пакет ARP;
- аппаратный адрес получателя (Target MAC Address) — MAC-адрес (дефис) устройства, которому назначен пакет ARP (в запросах ARP это поле обычно пустое);
- IP-адрес получателя (Target IP Address) — IP-адрес устройства, которому назначен пакет ARP.

Алгоритм работы протокола `ARP`:
- ARP-модуль по заданному IP-адресу ищет MAC-адрес в таблице ARP. Если в ней есть нужный адрес, то MAC-адрес возвращается сетевому модулю.
- Если нужная запись отсутствует, то ARP-модуль отправляет широковещательный кадр канального уровня с ARP-запросом внутри. При этом в запросе указываются оба адреса отправителя (IP- и MAC-) и IP-адрес получателя.
- Все хосты получают ARP-запрос, но отвечает только тот хост, чей IP-адрес указан в поле IP-адреса получателя.
- ARP-модуль получателя проверяет тип оборудования и тип протокола. Если типы оборудования поддерживаются данным хостом, то хост:
- запоминает адреса отправителя в своём кеше;
- подставляет в ARP-ответ свой MAC-адрес, переставляя местами адреса отправителя и получателя.
- ARP-ответ направляется хосту, пославшему запрос, который помещает (или обновляет) полученную пару адресов в своём кеше.

Для повышения эффективности работы механизма ARP каждый хост при включении рассылает широковещательный запрос. В качестве IP-адреса получателя подставляется IP-адрес отправителя.
- Все хосты, получившие такой широковещательный запрос, запоминают адреса отправителя в своём кеше и не отвечают на запрос.
- В случае если хост-получатель обнаруживает, что его адрес совпадает с адресом хоста-отправителя, хост-отправитель извещается о дублирующем адресе.

Для обновления данных в кеше хост регулярно направляет одноадресный (unicast) запрос нужному хосту. Если хост несколько раз не отвечает на ARP-запрос, то соответствующая строка в таблице хоста-отправителя удаляется.

Чтобы объединить несколько канальных сетевых сегментов в один, используется Proxy ARP. Это модифицированный вариант ARP-протокола. Он используется шлюзами, которые находятся в нескольких сетях одновременно. У Proxy ARP информация о подключённых сетях хранится в одной объединённой ARP-таблице.

Команды `ip neigh` и `arp` позволяют просматривать, добавлять и удалять записи в ARP-таблице.

Команд arp может использоваться со следующими параметрами:
- -a — позволяет вывести содержимое кеша в компактном формате;
- -n — позволяет вывести IP-адреса узлов, даже если известны их имена;
- -s — используется для добавления элементов в кеш;
- -d — используется для удаления элементов из кеша;
- -f — позволяет добавить элементы в кеш из файла;
- -i — позволяет указать интересующий сетевой интерфейс.

Для добавления записи в ARP-таблицу следует использовать команду  `arp -s <ip_address> <mac_address>`  где  `<ip_address>`  — это IP-адрес узла, запись которого вы хотите добавить;  `<mac_address>`  — это MAC-адрес узла, запись которого вы хотите добавить.

> Например,  `arp -s 192.168.1.1 00:11:22:33:44:55`  — команда добавляет в ARP-таблицу запись, указывающую, что IP-адрес  `192.168.1.1`  соответствует MAC-адресу  `00:11:22:33:44:55` .

Для работы с командой требуются права суперпользователя.

Для удаления записи из ARP-кеша вы можете использовать следующую команду:  `arp -d <ip_address>`  где  `<ip_address>`  — это IP-адрес узла, запись которого вы хотите удалить.

> Например,  `arp -d 192.168.1.1`  — команда удалит из ARP-таблицы запись для IP-адреса  `192.168.1.1` .

## IP (Internet Protocol) и внедрение меток безопасности

IP-датаграмма состоит из заголовка и данных. Данные поступают (**инкапсулируются**) от протоколов верхнего уровня (`TCP`, `UDP`, `ICMP` и др.). Если размер IP-датаграммы больше, чем максимальный размер данных в кадре канального уровня (`MTU`, Maximum Transmission Unit), то IP-датаграмма разбивается на IP-фрагменты для последующей инкапсуляции в кадры канального уровня.
- MTU для Ethernet II — 1500 байт, 802.3 — 1492 байта.
- Современные Ethernet-адаптеры и коммутаторы поддерживают так называемые Jumbo-кадры, в которых MTU может принимать значения от 1500 до 16000 байтов. Как правило, MTU в Jumbo-кадрах не превышает 9000 байтов.

Каждый IP-фрагмент содержит стандартный IP-заголовок (20 байт). Фрагменты собираются в IP-датаграмму на хосте-получателе. IP-заголовок состоит из стандартного заголовка фиксированной длины (20 байт) и заголовка переменной длины (поле «Опции» (Options)). Максимальная длина IP-заголовка — 60 байт. Стандартный IP-заголовок содержит следующие поля:
- Версия (Version) — 4 бита, для IPv4 — 4.
- Длина заголовка (Internet Header Length, IHL) — 4 бита.
- Тип службы (Type of Service, ToS) — 1 байт.
- Длина IP-датаграммы (Total Length) — 2 байта.
- Уникальный идентификатор датаграммы (Identification) — 2 байта.
- Флаги (Flags) — 3 бита. 17-й бит запрещает/разрешает фрагментацию IP-датаграммы, 18-й бит указывает, что фрагмент является последним (MF=0).
- Смещение фрагмента, которое определяет положение фрагмента в IP-датаграмме (Fragment Offset), — 13 бит.
- Время жизни, определяющее количество прыжков, которое разрешено пройти IP-датаграмме до хоста-получателя (TTL, Time To Live), — 1 байт.

Начальное значение TTL в IP-заголовке регулируется параметром ядра. Каждый маршрутизатор уменьшает TTL на единицу.

- Код протокола (Protocol) — 1 байт. `TCP` — `6`, `UDP` — `17`, `ICMP` — `1`.
- Контрольная сумма IP-заголовка (Header Checksum) — 2 байта.
- Адрес отправителя (Source Address) — 4 байта.
- Адрес получателя (Destination Address) — 4 байта.

При работе в Astra Linux на уровнях конфиденциальности и при наличии категории, подразумевающей защиту информации и недопустимости её получения несанкционированными получателями, передаваемая в сеть информация должна маркироваться с учётом её принадлежности тому или иному уровню конфиденциальности и категории. Для этого используются классификационные метки.

Классификационные метки (уровни конфиденциальности и категории) помещаются в дополнительную часть IP-заголовка в поле «`Опции`» с типом «`Безопасность`» (Basic Security Option). Общие требования к использованию опций безопасности в IP-заголовке изложены в RFC 1108 Security Options for the Internet Protocol. Формат и правила установки классификационных меток в IPv4-сетях, используемых в системах с установленной ОС Astra Linux, описываются в стандарте ГОСТ Р 58256-2018 «Управление потоками информации в информационной системе. Формат классификационных меток».

Классификационная метка — это служебный атрибут безопасности единицы информационного ресурса, представляющий собой комбинацию иерархических классификационных уровней (степеней секретности) и неиерархических классификационных признаков (категорий).

В случае отсутствия в заголовке IP-датаграммы опции безопасности считается, что информация передаётся с нулевым уровнем конфиденциальности и без категорий.

## TCP (Transmission Control Protocol)

Протокол TCP (RFC 793, 1122, 3168, 6093, 6528) ориентирован на установление соединения между сетевыми процессами и надёжную доставку данных от одного сетевого процесса другому. Модуль TCP-протокола выполняет следующие функции:
- принимает данные из модулей протоколов верхнего прикладного уровня, при этом данные воспринимаются как неструктурированный поток байтов;
- разбивает данные на сегменты;
- добавляет в каждый сегмент TCP-заголовок;
- отправляет TCP-сегменты в модуль IP-протокола.

Для идентификации сетевых процессов на хосте используются номера портов. В TCP-заголовок помещаются номер порта отправителя и номер порта получателя.

Порты классифицируются следующим образом (rfc6335):
- общеизвестные (системные) (0−1023) — зарезервированы IANA для популярных сетевых служб;
- зарегистрированные (пользовательские) (1024−49151) — используются для сетевых служб, данные порты выделяются и регистрируются IANA;
- динамические (49152−65535) — используются клиентскими сетевыми процессами.

Соответствия имён сетевых служб и номеров портов хранятся в файле `/etc/services`.

Установление TCP-соединения (сокета) происходит путём троекратного обмена специальными сегментами, использующими флаги `SYN` и `ACK`, которые устанавливаются в TCP-заголовке (процедура тройного рукопожатия). Для завершения TCP-соединения используются специальные сегменты с установленными в заголовках флагами `FIN` и `ACK`.

Надёжность доставки TCP-сегментов обеспечивается:
- подтверждением принятия данных TCP-модулем получателя (TCP-модуль получателя посылает TCP-модулю отправителя подтверждающий TCP-сегмент с установленным флагом ACK);
- повторной отправкой TCP-сегмента сетевым процессом-отправителем в случае непоступления подтверждающего TCP-сегмента (при отправке сегмента TCP-модуль отправителя запускается таймер ожидания поступления подтверждающего сегмента; если время тайм-аута истекло, отправка сегмента делается повторно);
- упорядочиванием поступающих TCP-сегментов;
- отбрасыванием дублирующих сегментов;
- внедрением контрольной суммы TCP-сегмента (вместе с заголовком) в TCP-заголовок и отбрасыванием повреждённых сегментов.

TCP-протокол поддерживает управление потоками данных с помощью механизма скользящего окна, который регулирует скорость передачи данных так, чтобы не перегрузить принимающую сторону.

TCP-модуль может использоваться несколькими прикладными процессами (мультиплексирование). У каждого соединения TCP есть два конца, каждый из которых идентифицируется четырьмя элементами: IP-адрес и порт отправителя, IP-адрес и порт получателя. Это позволяет одному хосту установить множество соединений TCP с другим хостом, используя разные порты. TCP использует понятие **портов** для **идентификации разных приложений или сессий внутри одного и того же соединения**. Таким образом, данные от разных приложений или сессий и для них могут мультиплексироваться через одно и то же TCP-соединение. Когда данные приходят на TCP-соединение, TCP на уровне получателя демультиплексирует потоки данных обратно в их первоначальные сессии или потоки, основываясь на порте назначения.

Мультиплексирование является важной частью обработки TCP, потому что оно обеспечивает эффективное использование соединений, между тем как демультиплексирование обеспечивает корректное разделение данных на приёмной стороне.

## UDP (User Datagram Protocol)

Протокол UDP (User Datagram Protocol, RFC 768) используется в сетевых коммуникациях для передачи коротких сообщений или датаграмм от одного узла к другому без необходимости установления подключения и подтверждения доставки. Это делает UDP идеальным для приложений, где скорость важнее надёжности или порядка получения данных. Протокол UDP (так же, как и TCP) может различать прикладные процессы по номерам портов. UDP поддерживается проверка контрольной суммы данных. Он быстрее и эффективней, чем TCP, но уступает в надёжности. Также могут возникать проблемы с перегрузкой.

## ICMP (Internet Control Message Protocol)

Протокол ICMP (Internet Control Message Protocol, RFC 792) предназначен для обмена информацией сетевого уровня между хостами, как правило, для передачи сообщений об ошибках и исключительных ситуациях. Он работает поверх IP-протокола, помещая ICMP-сообщения внутрь IP-пакета.

Наиболее часто используются следующие комбинации типов и кодов:
- **тип=0,** код=0 — эхо-ответ на команду ping (`echo reply`);
- **тип=3**, код=0 — сеть-адресат недоступен (`network unreachable`);
- **тип=3**, код=1 — хост-получатель недоступен (`host unreachable`);
- **тип=3**, код=2 — протокол адресата недоступен (`protocol unreachable`);
- **тип=3**, код=3 — порт адресата недоступен (`port unreachable`);
- **тип=3**, код=4 — пакет уничтожен, поскольку требуется фрагментация, но флаг DF=1;
- **тип=3**, код=6 — сеть-адресат неизвестен (`network unknown`);
- **тип=3**, код=7 — хост-получатель неизвестен (`host unknown`);
- **тип=8**, код=0 — запрос эха (`echo request`);
- **тип=11**, код=0 — время жизни пакета истекло (`time exceeded`);
- **тип=12**, код=0 — неверный IP-заголовок (`parameter problem`).

## Настройка сетевых интерфейсов

Если установлено графическое окружение, то все сетевые интерфейсы, кроме loopback (lo), находятся по умолчанию под управлением NetworkManager, за запуск которого отвечает служба NetworkManager.service. Изменять сетевые настройки может обычный пользователь, права root не требуются. Конфигурационный файл расположен по следующему пути:

```
/etc/NetworkManager/NetworkManager.conf
```

Файлы с информацией о настроенных сетевых соединениях расположены по пути:

```
/etc/NetworkManager/system-connections/
```

Для их просмотра потребуются права `root`. Просмотр информации о сетевых соединениях может выполняться без прав суперпользователя с использованием утилит.

Если графическое окружение не установлено, то сетевые интерфейсы инициализируются командами `ifup/ifdown`, за работу которых отвечает служба `networking.service`. В случае установленного графического окружения `ifup/ifdown`-команды инициализируют только интерфейс `loopback (lo)`. Основной конфигурационный файл расположен по следующему пути:

```
/etc/network/interfaces
```

Для настройки сетевых интерфейсов также может использоваться внутренняя подсистема управления сетевыми интерфейсами `systemd`. За это отвечает служба `systemd-networkd.service`, которая по умолчанию неактивна.

Для просмотра и ручной временной настройки сетевых интерфейсов может быть использована команда `ip`:
- просмотр настроек сетевых интерфейсов осуществляется командами:
	- `ip address show`
	- `ip -a`

- изменение настроек сетевых интерфейсов производится с помощью команд:
	- `ip address add A.B.C.D/M dev ethX`
	- `ip ethX A.B.C.D/M`

## Настройка с помощью ifup/ifdown

В случае если настройка сетевых интерфейсов будет производиться через `ifup/ifdown`-команды, то рекомендуется отключить службу `NetworkManager`:
- `systemctl mask NetworkManager` — блокирует запуск `NetworkManager`;
- `systemctl stop NetworkManager` — останавливает работу службы `NetworkManager`;

При этом управление сетью производится службой `networking.service`.

Управление сетевыми соединениями выполняется следующими командами:
- `ifup` — активирует соединение;
- `ifdown` — деактивирует соединение;
- `ifquery` — отображает информацию о соединениях.

Все команды выполняются от имени суперпользователя.

Можно заметить, что в списке отсутствуют команды для указания параметров соединения. Дело в том, что параметры соединения задаются в файле `/etc/network/interfaces`. Подробное описание того, как заполнять этот файл, указано в man interfaces. Основными являются следующие параметры:
- `auto ethX` — интерфейс `ethX` будет активирован при загрузке;
- `iface ethX inet dhcp` — на интерфейсе `ethX` будет использоваться динамическая адресация;
- `iface ethX inet static` — на интерфейсе `ethX` будет использоваться статическая адресация. При указании этого параметра также следует указать следующие:
	- address A.B.C.D/M — адрес/маска;
	- gateway А.B.C.D — шлюз

Пример настройки `eth0` для динамической адресации и `eth1` — для статической (IPv4 и IPv6):

```shell
auto eth0
allow-hotplug eth1
iface eth0 inet dhcp
iface eth0 inet6 auto
iface eth1 inet static
	address 192.168.1.2/24
	gateway 192.168.1.1
iface eth1 inet6 static
	address fec0:0:0:1::2/64
	gateway fec0:0:0:1::1
```

## Агрегирование (объединение) интерфейсов Ethernet

**Агрегирование сетевых интерфейсов** — это объединение нескольких физических интерфейсов в один логический.

Агрегирование сетевых интерфейсов позволяет:
- увеличить пропускную способность (но не скорость!) каналов передачи данных за счёт различных методов балансировки нагрузки;
- повысить отказоустойчивость.

Основные режимы агрегирования:
- `balance-rr (0)` — отправка сетевых пакетов поочерёдно через все агрегированные интерфейсы (политика round-robin) — баланс., отказ.
- `active-backup (1)` — отправка сетевых пакетов через активный интерфейс. При отказе активного интерфейса (link down и т. д.) автоматически переключается на резервный интерфейс; — отказ.
- `balance-xor (2)` — каждый интерфейс работает со своим получателем — баланс., отказ.
- `balance-tlb (5)` — исходящий трафик распределяется, исходя из загруженности каждого интерфейса (transmit load balancing);
- `alb (balance-6)` — входящий и исходящий трафики распределяются, исходя из загруженности каждого интерфейса (adaptive load balancing).

Алгоритм создания `bonding` с помощью `NetworkManager` (команда `nmcli`):
- создаём логический ведущий (master) интерфейс:

﻿ `nmcli con add type bond ifname имя_мастера` ﻿

- определяем подчинённые (slave) интерфейсы:

﻿ `nmcli con add type ethernet ifname ethX master имя_мастера` ﻿

﻿ `nmcli con add type ethernet ifname ethY master имя_мастера` ﻿

- активируем соединения, соответствующие подчинённым интерфейсам:

﻿ `nmcli con up bond-slave-ethX` ﻿

﻿ `nmcli con up bond-slave-ethY`  ﻿

Изменение поведения `bonding` производится с помощью параметра `bond.options`:

﻿ `nmcli con mod bond-имя_мастера bond.options mode=active-backup` ﻿

﻿ `nmcli con up bond-имя_мастера` ﻿

Если требуется задать статический адрес соединению типа `bonding`, то это делается точно так же, как и для обычного соединения. В качестве имени соединения указываем `bond-имя_мастера`.

В случае настройки `bonding` через команды `ifup/ifdown` в файл `/etc/network/interfaces` помещаются настройки вида:

```
auto ethX 
iface ethX inet manual 
auto ethY 
iface ethY inet manual 
auto bond0 
iface bond0 inet static 
address A1.B1.C1.D1 
netmask M1.M2.M3.M4 
network A3.B3.C3.D3  
bond-slaves ethX ethY 
bond-mode active-backup 
bond-fail-over-mac 1 
bond_miimon 100 
bond_downdelay 200 
bond_updelay 200
```

Также следует установить пакет ifenslave, который обеспечит работу bond-интерфейса со службой networking из основного репозитория:

﻿ `apt install ifenslave` ﻿

Мониторинг поведения bonding осуществляется путём чтения файла `/proc/net/bonding/имя_мастера`:

﻿ `watch cat /proc/net/bonding/bond0` ﻿

Также за работой bond-интерфейсов можно следить с помощью утилиты `iftop`.

## Сетевые службы DNS и DHCP

Для управления адресом интерфейса по протоколу DHCP используется утилита `dhclient`.

Если адрес сетевому интерфейсу назначается динамически, то следует проверить доступность сервера DHCP командой

﻿ `sudo dhclient -v` ﻿

Для получения IP-адреса следует выполнить команду:

﻿ `sudo dhclient имя_интерфейса` ﻿

Для высвобождения адреса используется следующая команда:

﻿ `sudo dhclient -r имя_интерфейса` ﻿

Для проверки DNS используется универсальная утилита nslookup. Она поможет получить IP-адрес для символьного имени хоста, а также сделать обратное преобразование и определить скорость работы DNS-сервера.

По умолчанию утилита отсутствует в системе. Получить её можно из репозиториев в составе программного пакета `dnsutils`.

Рассмотрим синтаксис команды:

﻿ `nslookup параметры домен` ﻿

Основные параметры:
- `type` — тип записи DNS: ns, txt, ptr, cname, soa и др.;
- `port` — порт DNS;
- `recurse` — если DNS не отвечает, то использовать другие DNS;
- `retry` — количество попыток;
- `timeout` — время;
- `fail` — если DNS-сервер возвращает ошибку, то использовать другой сервер.

С её помощью можно получить различную информацию о доменных именах, такую как IP-адреса, записи `MX` (для электронной почты), записи `NS` (имена серверов имён) и многое другое. Это мощный инструмент для диагностики сетевых проблем, проверки настроек `DNS` и общего анализа доменных имён.

Рассмотрим синтаксис:

## Маршрутизация

**Маршрутизация** — процесс определения оптимального пути для передачи IP-пакета от отправителя к получателю через совокупность сетей (интернет). Устройства, которые выполняют эту задачу, называются маршрутизаторами (роутерами).

Для принятия решений о маршрутах каждый маршрутизатор использует таблицу маршрутизации. Именно способы заполнения этой таблицы и разделяют маршрутизацию на статическую и динамическую.

**Таблица маршрутизации** — это структура данных, хранящаяся в ядре и содержащая информацию о доступных путях в сети (`netstat -r`, `ip route`, `route`)

Два типа маршрутизации в IP-сетях:
- **статическая** — это процесс, при котором администратор сети вручную добавляет, изменяет или удаляет маршруты в таблице маршрутизации;
- **динамическая** (внутренняя (interior) и внешняя (exterior)) — это процесс, при котором маршрутизаторы автоматически обмениваются информацией о доступных сетях и вычисляют оптимальные пути с помощью специальных протоколов.

В корпоративных сетях используют протоколы внутренней динамической маршрутизации:
- `RIP` (Routing Information Protocol) — один из старейших протоколов. Его главный недостаток — в принципе определения оптимальности маршрутов только по длине перелёта пакетов, без учёта параметров каналов связи;
- `OSPF` (Open Shortest Path First) (bird, quagga) — был разработан как прямой ответ на недостатки `RIP`. Это мощный, быстрый и масштабируемый протокол.

## Диагностика уровня сетевого доступа

Первые шаги для устранения неполадок:
- Проверка сетевого оборудования, сетевых кабелей.
- Проверка цвета светодиода на сетевой карте Ethernet.
- Выполнить команду `ip link show`:
	- UP — сетевой интерфейс подсоединён;
	- NO-CARRIER — на сетевой интерфейс не приходит сигнал.

В Astra Linux используется по умолчанию традиционная схема именования сетевых Ethernet-интерфейсов: `eth0`, `eth1`, `…`

Команда `ip link show` имеет следующие полезные варианты использования:
- `ip -br link show` — краткая форма представления информации о состоянии интерфейсов;
- `ip -s link show` — показывает статистику переданных и полученных пакетов данных для разных интерфейсов.

Если команда `ip link` не показывает никакой информации обо всех сетевых интерфейсах (количество установленных сетевых карт больше количества сетевых интерфейсов), то следует:
- выполнить команду `lspci`, чтобы определить, распознало ли ядро ОС сетевую карту;
- установить и загрузить драйвер (модуль ядра), соответствующий установленной сетевой карте, с помощью команды

﻿ `sudo modprobe имя_драйвера` ﻿

Сетевые карты Ethernet поддерживают различные режимы передачи данных (скорость передачи данных, полудуплекс / полный дуплекс).

Для согласования режимов передачи данных между разными Ethernet-устройствами используется механизм самосогласования (`autonegotiation`).

Для просмотра разрешённых и установленных режимов передачи данных сетевых карт Ethernet используются команды `ethtool` или `mii-tool`.

Утилита `ethtool` позволяет:
- получить информацию о драйвере сетевой карты (параметр `-i`);
- изменить настройки сетевой карты (параметр `-s`);
- получить статистику об использовании сетевого интерфейса, включая информацию об ошибках (параметр `-S`).

## Сетевые настройки и маршрутизация

Первые шаги для устранения неполадок:

- Проверка сетевых настроек:
	- IP-адрес;
	- сетевая маска;
	- маршрут по умолчанию;
	- разрешение имён хостов (DNS).

- Способы настройки сетевых интерфейсов:
	- NetworkManager;
	- команды ifup/ifdown, файл /etc/network/interfaces.

- Динамическое и статическое назначение IP-адреса и сетевых настроек.

С помощью команды `ip address show` следует убедиться, что IP-адрес и сетевая маска настроены правильно.

Если обнаружены ошибки в значениях IP-адреса или сетевой маски, то следует:
- установить способ настройки сетевых интерфейсов (`NetworkManager` или команды `ifup/ifdown`);
- выяснить способ назначения IP-адреса (статический или динамический).

Если адрес сетевому интерфейсу назначается динамически (через DHCP), то следует проверить доступность сервера DHCP командой

﻿ `sudo dhclient -v` ﻿

Для определения способа настройки следует выяснить статус соответствующих служб `NetworkManager.service` и `networking.service `командой

﻿ `systemctl status имя_службы` ﻿

Если активна только служба `NetworkManager.service`, то необходимо исправить сетевые настройки с помощью любой из утилит:
- n`m-connection-editor` (графическая утилита);
- `nmtui` (псевдографическая утилита);
- `nmcli` (утилита командной строки).

Если активна только служба `networking.service`, исправить сетевые настройки в файле `/etc/network/interfaces` и проверить содержимое файлов в каталоге `/etc/network/interfaces.d`.

Если активны обе службы, то нужно убедиться, что под управлением NetworkManager находятся все интерфейсы, кроме интерфейса `loopback`. Выполните команду

﻿ `nmcli | grep unmanaged` ﻿

и обратите внимание, у каких сетевых интерфейсов стоит отметка без управления (unmanaged).

Если какой-то из сетевых интерфейсов (кроме `loopback`) находится не под управлением `NetworkManager`, то следует выбрать один из двух вариантов настройки:

- отключить службу NetworkManager.service:

﻿ `systemctl stop NetworkManager.service` ﻿

﻿ `systemctl mask NetworkManager.service` ﻿

и настроить все интерфейсы в файле `/etc/network/interfaces`;

- настроить все интерфейсы (кроме loopback) через `NetworkManager`.

На любом хосте, подключённом к сети, должен быть настроен маршрут (шлюз) по умолчанию. Проверку параметров такого маршрута (прежде всего IP-адрес шлюза) можно осуществить командой `ip route show`.

Следует убедиться, что шлюз по умолчанию находится в одной из сетей, непосредственно подключённых к хосту.

Временно исправить шлюз по умолчанию можно командой

﻿ `ip route add default via IP_адрес_шлюза` ﻿

Для изменения маршрута по умолчанию следует:
- использовать утилиты для `NetworkManager`, если интерфейсы управляются этой службой;
- указать параметр gateway IP_адрес в файле `/etc/network/interfaces`.

Настройку DNS можно проверить путём просмотра содержимого файла `/etc/resolv.conf`.

Если адреса DNS-серверов указаны неверно, то следует:
- использовать утилиты для `NetworkManager`, если интерфейсы управляются этой службой;
- исправить адреса серверов DNS в файле `/etc/resolv.conf`, если интерфейсы настраиваются в `/etc/network/interfaces`.

## Доступность хостов

Первые шаги для устранения неполадок:
- Проверка доступности хостов по IP-адресу или DNS-имени:
	- работоспособность стека TCP/IP на локальном хосте;
	- доступность шлюза;
	- доступность удалённого хоста.

- Утилиты `ping` и `traceroute`.

Утилита `ping` используется главным образом для проверки доступности хостов в локальной сети (сети предприятия), поскольку принято блокировать ICMP-сообщения на периметре сети предприятия.

Также `ping` может использоваться для оценки качества линии связи.

Порядок использования команды `ping`:
- `ping localhost` или `ping 127.0.0.1` — проверка работоспособности локального стека `TCP/IP`;
- `ping IP_адреса_шлюза` — проверка доступности хоста в локальной сети и шлюза по умолчанию.

Для оценки качества линии связи с помощью команды `ping` следует использовать ICMP-сообщения большого размера (параметр `-s`) и обратить внимание на количество потерянных пакетов в сводке, которая приводится в конце вывода команды.

![[Pasted image 20260520150727.png]]

Утилита `traceroute` используется в основном для проверки доступности хостов, находящихся вне локальной сети, и определения возможных проблем с маршрутизацией за пределами сети предприятия.

![[Pasted image 20260520150831.png]]

Также `traceroute` по умолчанию использует протокол `UDP`. Поэтому пакеты, отсылаемые данной утилитой, реже отфильтровываются внешними маршрутизаторами.

## Доступность сетевых служб удалённых хостов

Первые шаги для устранения неполадок:

- Протоколы `TCP` и `UDP`:
	- порты,
	- сокеты.

- Сканирование портов удалённых хостов:
	- nc,
	- nmap.

Для адресации сетевых процессов на хосте используются номера портов. Номер порта может принимать значение от `0` до `65535`.

Для серверных сетевых процессов порты выделяются статически, а порты для клиентских серверных процессов — динамически из диапазона, определённого в параметре ядра `/proc/sys/net/ipv4/ip_local_port_range` (по умолчанию — 32768–60999).

Порты определяются на транспортном уровне модели `TCP/IP` (протоколы `TCP` и `UDP`).

Сетевой сокет — это программный интерфейс, который предназначен для обмена данными между процессами и определяется IP-адресом хоста и портом, соответствующим процессу на данном хосте.

Сетевое сокетное соединение — это связь двух сокетов: сокета отправителя (клиента) и сокета получателя (сервера).

Серверные сокеты вначале находятся в состоянии прослушивания (listening), ожидая, когда клиентский сокет отправит запрос.

Доступность удалённых портов (удалённых сетевых служб) может быть проверена командами:
- `ncat` (nc) — программный пакет `ncat` из `extended-репозитория`;
- `nmap` — программный пакет `nmap` из `extended-репозитория`.

Утилита `ncat` имеет ограниченную функциональность по сканированию портов:
- проверка доступности удалённой службы:

﻿ `ncat адрес_хоста порт` ﻿

- сканирование портов:

﻿ `ncat -zv адрес_хоста диапазон_портов` ﻿

Утилита `nmap` предназначена для сканирования портов и поддерживает разные режимы сканирования:
- `-sT` — пытается установить TCP-соединение (выполнить процедуру тройного рукопожатия);
- `-sS` — отправляется пакет с установленным флагом `SYN` (используется по умолчанию, самый популярный режим сканирования);
- `-sA` — отправляется пакет с установленным флагом `ACK`; не может определить открытые порты, но хорошо умеет определять порты, отфильтрованные межсетевыми экранами;
- `-sU` — сканируются UDP-порты.

![[Pasted image 20260520151107.png]]

Утилита `nmap` показывает следующие состояния портов:
- `open` — процесс «слушает» на этом порте, ожидая запрос от клиента;
- `filtered` — порт заблокирован межсетевым экраном;
- `closed` — процесс не «слушает» в данный момент, но может возобновить свою работу;
- `unfiltered` — nmap не может определить, находится ли порт в состоянии open или closed.

## Обнаружение и устранение неисправностей в работе сетевых служб

Первые шаги для устранения неполадок:

- Проверка работы локальных сетевых служб:
	- ﻿ `netstat -tlnp`  или  `ss -tlnp` ﻿
	- ﻿ `journalctl -u имя_службы.service` ﻿
- Обнаружение DoS-атак:
	- SYN Flood атака  `watch "netstat -tan | grep -i syn | wc -l"` ﻿

После запуска сетевой службы следует убедиться, что сетевой процесс стартовал и ему назначен соответствующий порт: будет показана информация о TCP- (параметр -t) и UDP-портах (параметр -u) и процессах (параметр -p), «слушающих» (параметр -l) на этих портах:

﻿ `sudo netstat -nltup или sudo ss -nltup`

![[Pasted image 20260520151227.png]]

При просмотре вывода команды netstat или ss следует обратить внимание на то, на каком интерфейсе была запущена служба:
- если указан адрес `127.0.0.1`, то служба будет доступна только локально;
- если указан адрес `0.0.0.0`, то служба будет работать на всех сетевых интерфейсах хоста.

В случае если сетевая служба не стартовала или не использует требуемый сетевой интерфейс, следует проанализировать сообщения из журнала:

﻿ `journalctl -u имя_юнита_службы.service` ﻿

Одним из распространённых способов хакерских атак на сетевые службы является так называемая DoS-атака (Denial of Service, отказ в обслуживании).

DoS-атака, в частности, может быть реализована с помощью отправки с различных поддельных хостов большого количества пакетов, в которых установлен флаг `SYN`, — начальная стадия установления TCP-соединения (`SYN Flood` атака).

Обработка большого количества SYN-пакетов и существование большого количества полуоткрытых соединений могут существенно нагрузить вычислительные ресурсы сервера (процессор, память), что будет приводить к замедлению работы сервера вплоть до состояния невозможности дальнейшего обслуживания запросов клиентов.

Количество полуоткрытых соединений можно посчитать командой

﻿ `netstat -tan | grep -i syn | wc -l` ﻿

Большое количество таких соединений может означать, что идёт SYN Flood атака.

## Поиск и решение проблем на прикладном уровне

Первые шаги для устранения неполадок:
- анализ перехваченного сетевого трафика — tcpdump;
- анализ журнальных файлов;
- запуск служб в режиме отладки (debug);
- диагностика неисправностей с помощью специфических инструментов для разных протоколов прикладного уровня:
	- DNS: nslookup, dig;
	- DHCP: dhclient;
	- HTTP/HTTPS: curl, wget;
	- SMTP: nc адрес_сервера 25.

Самым универсальным средством для выявления проблем в функционировании сетевых процессов, работающих на прикладном уровне модели TCP/IP, является перехват и анализ трафика между клиентом и сервером.

Два наиболее популярных средства для перехвата трафика:
- `tcpdump` — программный пакет из репозитория `base`;
- `wireshark` — программный пакет из репозитория `base`.

Утилита `wireshark` имеет графический интерфейс и поддерживает интерпретацию данных для огромного количества протоколов (Analyze — Enabled Protocols).

Основные параметры утилиты `tcpdump`:
- `-i имя_интерфейса` — указывает интерфейс, на котором будет перехватываться трафик (any — все интерфейсы);
- `-w имя_файла` — сохранение перехваченного трафика в файл;
- `-r имя_файла` — чтение перехваченного трафика из файла;
- `-с число` — количество записей о трафике, которые будут перехвачены;
- `-n` — не разрешать имена (крайне рекомендуется для постоянного использования);

Основные фильтры (BPF, Berkeley Packet Filter) для утилиты `tcpdump` (man pcap-filter):
- `host` — фильтрация по адресу или имени хоста (перед host можно поместить scr или dst, указывая направление трафика);
- `net` — фильтрация по номеру сети (перед `net` можно поместить `src` или `dst`);
- `port` — фильтрация по номеру или имени порта (имя берётся из `/etc/services`, перед `port` можно поместить `scr` или `dst`);
- имя протокола: `tcp`, `udp`, `icmp`, `arp`, `ip`, `ipv6`;
- фильтры для заголовков протоколов:` имя_протокола[байт_заголовка]`. Примеры: `icmp[0]` — байт, хранящий тип ICMP-сообщения, `tcp[13]` — флаги TCP.

При составлении фильтров можно использовать логические операции `and`, `or`, `not` и группировку условий отбора с помощью круглых скобок.

Пример: выполнены следующие действия:
- перехватываются 10000 записей, которые записываются в файл traffic.pcap;
- вычисляется количество полученных веб-сервером (192.168.100.105) SYN-пакетов — 5187;
- вычисляется количество полученных веб-сервером ACK-пакетов — 0;
- вычисляется количество отправленных веб-сервером SYN/ACK-пакетов — 4835.

## Протокол Secure Shell

Протокол Secure Shell (SSH) — это протокол для безопасного входа в систему удалённого доступа и других безопасных сетевых сервисов по небезопасной сети. Весь трафик (включая пароли) шифруется, уменьшая риск перехвата. SSH можно использовать для передачи любого типа сетевых служб, как то `FTP`, `POP`, `IMAP` и др., а также для создания безопасных туннелей для различных типов трафика, обеспечивая дополнительные слои безопасности.

Для создания SSH-соединения требуется два компонента:
- служба `sshd` — сервер, к которому требуется подключиться;
- служба `ssh` — клиент, который будет выполнять подключение.

Сервер прослушивает 22-й порт (по умолчанию) и при успешной аутентификации даёт доступ пользователю. Все команды, которые вводит пользователь на клиенте, отправляются через защищённый канал связи на сервер, на котором они выполняются и с которого отправляют результат работы обратно клиенту.

Управление службой ssh/sshd осуществляется с помощью системной утилиты Systemctl. Она используется в операционной системе Linux для управления службами или демонами. В таблице приведены некоторые команды утилиты Systemctl и их назначение.

| Команда   | Назначение                                 |
| --------- | ------------------------------------------ |
| `start`   | Запустить службу                           |
| `status`  | Узнать состояние службы                    |
| `stop`    | Остановить службу                          |
| `restart` | Выполнить перезапуск службы                |
| `reload`  | Перечитать конфигурацию службы             |
| `enable`  | Включить автозагрузку службы               |
| `disable` | Выключить автозагрузку службы              |
| `show`    | Просмотреть параметры управления службой   |
| `revert`  | Отменить все изменения конфигурации службы |

## Настройка службы sshd

Чтобы узнать, запущена ли служба sshd на сервере, необходимо в терминале выполнить команду  `systemctl status sshd.service` .

Также это можно узнать через графическую утилиту «Системный монитор».

Если служба sshd не была установлена при установке системы, то можно установить её с помощью команды  
`sudo apt install openssh-server -y` .

Конфигурационный файл для службы sshd располагается по адресу `/etc/ssh/sshd_config`. Многие параметры закомментированы, но они указывают на значение по умолчанию, которое всё равно используется. Если требуется установить другие значения, то нужно раскомментировать строку с нужным параметром (убрать символ #) и внести изменения. Вот некоторые возможности в настройке службы sshd (для получения справки по настройке службы можно воспользоваться командой  `man 5 sshd_config` ):
- указать, на каком TCP-порте работает служба sshd;
- изменить разрешение на аутентификацию по паролю;
- изменить разрешение на удалённый запуск графических приложений, отображая графику на клиентских машинах с помощью клиентского X11-сервера;
- настроить использование механизмов аутентификации и поддержки сеансов PAM.

Для внесения изменения потребуются права суперпользователя. Чтобы изменения вступили в силу, можно использовать команду  `systemctl reload sshd.service` , которая выполнит перечитывание настроек службы без её перезапуска. Для перезапуска службы следует выполнить команду  `systemctl restart sshd.service` .

Чтобы повысить безопасность, рекомендуется изменить некоторые значения по умолчанию:
- `Port` — номер порта, который слушает сервис (22), — изменить значение на любое другое и в дальнейшем использовать этот номер порта;
- `MaxAuthTries` — количество попыток подключения (`6`) — уменьшить, например, до `3`;
- `LoginGraceTime` — время, дающееся для подключения (`2m`), — уменьшить, например, до `30`s;
- `PermitRootLogin` — разрешить вход пользователя root — установить значение `no` для запрета прямого входа под учётной записью `root`.

Дополнительно можно ограничить IP-адреса, с которых возможно подключение, например:
- в файле `/etc/hosts.deny` запретить все подключения:
```
SSHD: ALL
```
- в файле `/etc/hosts.allow` разрешить только необходимые подключения:
```
SSHD: 192.168.1.1
SSHD: 192.168.0.0/16
```

## SSH Использование основных команд

Работу с удалённым компьютером можно условно разделить на три этапа:

1. Подключение к удалённому компьютеру через ssh.
2. Решение оперативных/текущих задач.
3. Завершение работы с удалённым компьютером.

Для подключения к удалённому компьютеру необходимо выполнить команду типа

﻿  `ssh <имя_пользователя>@<имя_или_адрес_сервера>` ﻿

Если требуется подключение по нестандартному порту, то команда будет вида

﻿ `ssh -p ***** <имя_пользователя>@<имя_или_адрес_сервера>`  ﻿

где ***** — номер используемого для подключения порта.

При успешном подключении в строке терминала имя пользователя и имя компьютера клиента сменятся на имя пользователя и имя компьютера сервера.

Для завершения работы с удалённым компьютером достаточно ввести команду `exit`.

Для взаимодействия с удалённым компьютером предусмотрены специальные команды. Чтобы копировать файл на удалённую машину, нужно использовать команду вида:

﻿ `scp <имя_файла> <имя_пользователя>@<имя_или_адрес_сервера>:<имя_файла>` ﻿

Команда для копирования файла на локальную машину имеет вид:

﻿ `scp <имя_пользователя>@<имя_или_адрес_сервера>:<имя_файла> <имя_файла>` ﻿

В обоих примерах первое <имя_файла> — это путь и имя копируемого файла, а второе <имя_файла> — то, с каким именем будет файл скопирован (в первом случае — на удалённом компьютере, а во втором — на локальном компьютере). Если путь не указан, то копирование будет выполнено из домашнего каталога в домашний каталог.

Для монтирования удалённой файловой систем по ssh в локальный требуется выполнить команду вида:

﻿ `sshfs <имя_пользователя>@<имя_или_адрес_сервера>: ~/<точка_монтирования>` ﻿

В результате выполнения показанного примера домашний каталог пользователя на удалённом компьютере был монтирован в папку tmp домашнего каталога текущего пользователя.

Чтобы размонтировать подключенную файловую систему, следует выполнить команду вида:

﻿ `fusermount -u ~/<точка_монтирования>` ﻿

При необходимости выполнения команды на удалённом компьютере используется команда вида:

﻿ `ssh <имя_пользователя>@<имя_или_адрес_сервера> <команда>`

## Настройка аутентификации по ключам

Аутентификация по ключам в SSH предоставляет ряд преимуществ перед обычной аутентификацией по паролю:
- взломать ключ гораздо сложнее, чем сгенерированный пароль, поскольку ключи обычно имеют длину в сотни бит и создаются случайным образом;
- нет необходимости вводить пароль при каждом соединении с сервером (полезно для автоматизации заданий, например, для резервного копирования на удаленном сервере);
- возможность иметь несколько пар ключей для одного пользователя, каждую из которых можно ассоциировать с определённым устройством или заданием;
- можно настроить систему таким образом, чтобы требовалось и владение приватным ключом, и знание пароля (повышает уровень безопасности).

При установке службы `ssh` происходит автоматическая генерация приватного и публичного ключей. Ключи на клиенте хранятся в каталоге `~/.ssh` в файлах:
- `id_rsa` — приватный ключ пользователя;
- `id_rsa.pub` — публичный ключ пользователя;
- `known_hosts` — публичные ключи ssh-серверов.

Все публичные ключи, которым доверяет сервер, хранятся в файле `~/.ssh/authorized_keys`. Публичный ключ пользователя c клиента передаётся на сервер и добавляется в файл `authorized_keys`.

Справка по форматам `authorized_keys` и `known_hosts` содержится в соответствующих разделах man 8 sshd.

Чтобы выполнить настройку аутентификации по ключам, необходимо на стороне клиента сгенерировать свою пару ключей. Для этого нужно выполнить команду `ssh-keygen`, её можно указать без параметров. Далее следует согласиться с предложением записать создаваемый ключ в файл по умолчанию `~/.ssh/id_rsa`. На дальнейшие вопросы можно нажать `Enter`.

Параметр `-b` утилиты `ssh-keygen` позволяет указать размер ключа в битах. Минимально рекомендованный размер ключа — `2048` бит, что является значением по умолчанию.

Пара из открытого и закрытого ключей будет автоматически создана в подкаталоге `.ssh` домашнего каталога пользователя (файлы `id_rsa` и `id_rsa.pub`).

Далее открытый (публичный) ключ требуется передать на сервер, к которому нужно подключаться. Для этого нужно выполнить команду `ssh-copy-id <имя_или_адрес_сервера>` . На запрос пароля следует ввести пароль пользователя удалённого компьютера, после чего ключ будет передан и установлен.

Если имя пользователя, от имени которого будет производиться подключение, не совпадает с именем текущего пользователя, то можно указать имя

﻿ `ssh-copy-id <имя_пользователя>@<имя_или_адрес_сервера>` .

> Для сохранения переданного ключа на удалённом сервере должен существовать домашний каталог пользователя. Если домашнего каталога нет, передать ключ не получится.

Теперь подключиться к серверу можно без ввода пароля — с указанием (по необходимости) имени пользователя:

﻿ `ssh <имя_пользователя>@<имя_или_адрес_сервера>` .

## Создание SOCKS5-канала

SSH поддерживает динамическую переадресацию портов, с помощью которой на локальном хосте можно создать сокет (программный интерфейс) прокси-сервера, работающего по протоколам SOCKS4/5. SOCKS5 (SOCKet Secure версии 5) — это протокол, который передаёт сетевые пакеты от сервера к клиенту через прокси-сервер. Он обеспечивает аутентификацию, чтобы только уполномоченные пользователи могли получить доступ к серверу.

Для создания SOCKS5-канала необходимо выполнить команду вида

﻿ `ssh -fN -D <адрес_хоста:порт> <имя_пользователя>@<имя_или_адрес_сервера>` , где

- `-f` — процесс ssh запускается в фоновом режиме;
- `-N` — не требуется выполнять удалённую команду;
- `-D` — используется динамическая переадресация портов.

Также можно использовать параметр `-C` для сжатия всех данных.
