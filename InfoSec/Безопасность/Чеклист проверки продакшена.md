## 1. **Инфраструктура**

Проверяем:

- открытые панели администрирования (Grafana, Kibana, Jenkins, RabbitMQ, MinIO)
- публичные S3/MinIO/Blob‑бакеты
- открытые базы данных (Mongo, Redis, Elasticsearch, PostgreSQL)
- открытые Docker/K8s API
- публичные метрики (`/metrics`, Prometheus)
- публичные health‑checks (`/health`, `/ready`, `/live`)

Это ТОП‑1 источник взломов.

## 2. **Секреты и конфигурации**

Проверяем:
- секреты в Git (ключи, токены, пароли)
- секреты в Docker‑образах
- секреты в переменных окружения, доступных через `/debug`
- секреты в логах
- секреты в CI/CD артефактах
- `.env` файлы в проде

Это ТОП‑2 источник компрометации.

## 3. **CI/CD**

Проверяем:
- возможность запускать произвольный код в pipeline
- доступ к секретам через артефакты
- открытые GitLab/GitHub runners
- небезопасные self‑hosted runners
- отсутствие изоляции job‑ов
- отсутствие проверки pull‑request’ов

CI/CD — один из самых опасных слоёв.

## 4. **Kubernetes**

Проверяем:
- открытый kube‑api
- возможность `kubectl exec` без ограничений
- контейнеры с root
- отсутствие NetworkPolicies
- доступ к service account токенам
- отсутствие PodSecurityStandards

K8s почти всегда неправильно настроен.

## 5. **API безопасность**

Проверяем:
- отсутствие проверки токена
- токены без TTL
- JWT без проверки подписи
- hardcoded API‑ключи
- отсутствие rate‑limit
- отсутствие проверки ролей

API — самый атакуемый слой.

## 6. **Бизнес‑логика**

Проверяем:
- можно ли менять цену товара
- можно ли отменять заказ после доставки
- можно ли использовать купон бесконечно
- можно ли обойти лимиты
- можно ли повторно использовать транзакцию

Это то, что ломает деньги, а не сервера.

## 7. **Кеширование**

Проверяем:
- CDN отдаёт приватные данные
- кешируются ответы с токенами
- кешируются персональные данные
- shared cache poisoning
- неправильные заголовки Cache‑Control

Кеш ломает приватность чаще, чем SQLi.

## 8. **Очереди и брокеры сообщений**

Проверяем:
- открытые RabbitMQ/Kafka/NATS
- отсутствие авторизации
- возможность читать чужие сообщения
- возможность публиковать сообщения
- доступ к internal topics

Брокеры часто считают «внутренними» и не защищают.

## 9. **Мониторинг и логирование**

Проверяем:
- логи содержат токены/пароли
- логи доступны публично
- нет алертов на подозрительные действия
- нет логов неуспешных входов
- нет логов админских действий

Без логов атаки не видно.

## 10. **Сеть и периметр**

Проверяем:
- открытые порты
- отсутствие firewall правил
- отсутствие сегментации
- доступ к внутренним сервисам снаружи
- SSRF‑точки, ведущие внутрь сети

Периметр — первое, что ломают.
